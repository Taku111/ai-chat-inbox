## The Ten Most Important Rules

These are the decisions most likely to be second-guessed during implementation. Do not deviate from them.

**1. The AI does NOT fire on every inbound message. It fires after 60 seconds of silence.**

The webhook handler (Section 15.1 Step 11) writes a `pendingAiRequests/{conversationId}` document with `executeAt = now + 60s` and returns. It does not call the AI. A Cloud Scheduler job (Section 12.4, `functions/src/processAiRequests.ts`) runs every minute, finds documents where `executeAt <= now`, and calls the AI then. This is the Firestore debounce pattern. The reason: three rapid messages from a parent should produce one AI reply using all three as context тАФ not three separate AI calls. Implement `setTimeout` in the webhook and you have implemented the wrong thing тАФ `setTimeout` cannot work across serverless function instances.

**2. The outbound message ID is generated by the client, not the server.**

When an agent taps Send: the client generates a UUID, uses it as both `messageId` and `idempotencyKey`, creates a local optimistic message with that ID and `status: 'sending'`, clears the input, and only then POSTs to `/api/messages/send` with the UUID in the body. The server writes the Firestore document using that exact UUID as the document ID. When `onSnapshot` returns the confirmed document, `useMessages` matches it by ID and removes the optimistic copy. If the server generates its own ID, the optimistic and confirmed versions cannot be matched and the agent sees their message appear twice. See Section 12.2тАУ12.3.

**3. `status: 'sending'` never exists in Firestore. It is client-only.**

The Firestore message document is written with `status: 'sent'` directly. The `'sending'` state lives only in the `optimisticMessages` array in `useMessages`. This avoids a second write to update `'sending' тЖТ 'sent'`. See Section 12.4.

**4. Idempotency uses atomic `create()`, not `get()` then `set()`.**

The original naive approach тАФ check if idempotency key exists, if not write it тАФ has a TOCTOU (time-of-check-time-of-use) race condition. Two simultaneous Twilio retries can both pass the check before either writes. Use `adminDb.collection(PROCESSED_WEBHOOKS).doc(key).create(...)` тАФ this throws `ALREADY_EXISTS` (gRPC code 6) atomically if the document exists. Catch code 6 and return 200 immediately. Catch any other error and rethrow. See Section 15.1 Step 4.

**5. The webhook must validate `Content-Type: application/x-www-form-urlencoded` before reading the body.**

Twilio always sends form-encoded. A spoofed request with `Content-Type: application/json` would produce empty URLSearchParams тАФ potentially producing a predictable HMAC signature that could be brute-forced. Reject non-form requests before any body reading. See Section 15.1 Step 0.

**6. `unreadCount` uses `FieldValue.increment()` тАФ never set directly.**

Two simultaneous inbound messages both reading `unreadCount = 0` and both writing `unreadCount = 1` produces a wrong count. `FieldValue.increment(1)` is atomic. The same applies to `agentUnreadCounts[agentUid]`, `messageCount`, and any other counter. Never read-then-write a counter. See Section 6.4.

**7. `onSnapshot` is used for Open/Pending conversations only. Resolved and All tabs use one-time `getDocs()`.**

A real-time listener on 500 resolved conversations costs reads on every change. Resolved conversations rarely change тАФ agents use that tab for lookup, not live monitoring. Open and Pending tabs need live updates. This single rule saves 60тАУ80% of Firestore read costs. See Section 10.1.

**8. `initializeFirestore` uses `try/catch`, not a `getApps().length` condition.**

`initializeFirestore` throws if called more than once for the same app instance. In Next.js HMR, the module re-executes. The correct pattern: call `initializeFirestore`, catch the error, fall back to `getFirestore`. See Section 5.2 тАФ the exact code is in the spec, copy it verbatim.

**9. The `pendingAiRequests` upsert uses `set(..., { merge: true })` тАФ not `update()`, not `set()` without merge.**

`update()` throws if the document does not exist (first message from a new contact). `set()` without merge overwrites all fields (loses `createdAt`). `set(..., { merge: true })` creates if missing, updates if present тАФ exactly what debouncing needs. See Section 12.3.

**10. Every `тЪая╕П SYSTEM DESIGN:` note must be read before implementing the section it appears in.**

There are over 40 of these in the spec. Each one is a one-paragraph explanation of why a specific decision was made and what happens if it is changed. They are not optional reading тАФ they are the documentation for the 40 hardest decisions in this codebase.

## Wiring That Is Implied But Not Spelled Out in the Spec

The spec describes components individually. Here is how several of them connect:

### `ChatWindow` coordinates `AISuggestionBar` and `MessageInput`

`ChatWindow` holds local state `editSuggestionText: string`. When the agent clicks [Edit] in `AISuggestionBar`, it calls `onEditSuggestion(text)` which sets this state in `ChatWindow`. `MessageInput` receives it as `initialValue`. After the agent sends, `ChatWindow` clears it.

```tsx
// Inside ChatWindow:
const [editSuggestionText, setEditSuggestionText] = useState('')

<AISuggestionBar
  conversationId={conversation.id}
  onEditSuggestion={(text) => setEditSuggestionText(text)}
/>
<MessageInput
  conversationId={conversation.id}
  initialValue={editSuggestionText}
  onAfterSend={() => setEditSuggestionText('')}
/>
```
